# MFS-GCL R108/R109 Results

**Date**: 2026-06-22
**Candidate**: Missing-Feature Stress GCL (MFS-GCL)
**Status**: `FAILED_SPLIT0_GATE`

## Motivation

MFS-GCL tests whether a BGRL encoder becomes more useful when it must align a
node-wise feature-missing online view to a clean teacher target. This differs
from the existing BGRL `DropFeatures` transform, which drops whole feature
columns globally; MFS drops each node-feature entry independently.

Two variants were implemented in `baselines/BGRL/train_npg_transductive.py`:

- `mfs_clean`: stressed online view aligns directly to clean teacher target.
- `mfs_aux`: standard BGRL remains the main loss; stressed online view adds a
  clean-target auxiliary loss.

The reproducer support is in `baselines/BGRL/reproduce_npg.py`.

## GPU / Evaluation Protocol

All R108/R109 runs used `--device=auto` and resolved to `cuda`. Training
forward/backward passes and representation extraction therefore use GPU.
The final node-classification probe remains the canonical CPU sklearn
logistic-regression protocol, because replacing it with a GPU classifier would
change the historical comparison protocol unless separately equivalence-tested.

## Novelty Check

The route has high novelty risk. Recent and nearby work already covers random
feature masking, feature-discrepancy-aware graph contrastive learning, masked
graph contrastive learning, and adversarial/missing-feature masking. The main
MFS distinction is clean-target BGRL alignment under node-wise entry dropout,
but this distinction is too narrow without strong empirical gains.

Closest risks:

- RFM / Rethinking Feature Augmentation in GCL: random feature masking for GCL.
- FDAGCL: feature discrepancy-aware graph contrastive learning.
- Masked / adversarial feature graph contrastive learning and graph masked
  autoencoder hybrids.

## Decision

`FAILED_SPLIT0_GATE`. Do not run WebKB, 10-split expansion, or external
baselines for MFS-GCL. Keep the code only as a negative control for
node-wise feature-missing stress objectives.

The main-method search remains open. The next candidate should avoid ordinary
feature masking / augmentation robustness unless it introduces a much stronger
label-free routing principle or a new target family.
